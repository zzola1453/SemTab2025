# 주차별 실습 보고서 — Cell Entity Annotation 시스템 개발

**과제명**: 테이블 셀 엔티티 어노테이션(CEA) 시스템 개발  
**핵심 문제**: CSV 테이블의 각 셀 값을 Knowledge Graph 엔티티 URI에 자동 매핑

---

## 1주차 — CEA 태스크 분석 및 핵심 난제 파악

### 실습 내용
- CEA(Cell Entity Annotation)의 정의와 입출력 구조를 문서와 데이터셋을 통해 심층 분석하였다. 입력은 CSV 테이블과 타겟 셀 좌표(row_id, col_id, cell_value)이고 출력은 Wikidata 엔티티 URI(QID)이며, 평가 지표인 F1(Precision·Recall 조화평균)의 수식과 NIL 처리 방식(미제출 시 해당 셀 제외)을 정확히 이해하고 직접 계산 예시를 작성하였다.
- 데이터셋 826개 테이블·84,907개 타겟 셀 전체를 탐색 스크립트로 분석하여 컬럼 분포(col0·col1에 56% 이상 집중), 도메인 다양성(영화·인물·국가·조약 등), 타겟 셀 94.7%가 텍스트 엔티티 컬럼에 집중됨을 확인하고, 날짜·숫자 컬럼을 사전 필터링하면 처리 대상을 약 5% 줄일 수 있음을 수치로 검증하였다.
- CEA의 핵심 난제 6가지(Disambiguation, Homonymy, Alias Resolution, NIL Detection, Noise Robustness, Collective Inference)를 실제 데이터 사례와 함께 정리하였다. 예를 들어 "Brazil"이 국가(Q155)인지 영화(Q105512)인지는 셀 값만으로 판단 불가능하며 컬럼 구조와 이웃 셀이 결정적 단서가 됨을 구체적 예시로 확인하였다.
- 선행 연구 논문 2편(RAGDify, Iterative Refinement)을 확보하여 각 접근법의 파이프라인 구조, 사용한 검색 전략, LLM 활용 방식, 성능 수치를 비교 정리하고, 두 논문에서 공통적으로 테이블을 셀 단위가 아닌 전체 단위로 처리한다는 설계 원칙을 도출하여 우리 시스템의 방향성을 수립하였다.

### 결과
- CEA 문제를 단순 키워드 매칭이 아닌 **문맥 기반 의미 해석 문제**로 재정의함. 같은 "Jordan"이라는 셀 값이 인물인지 국가인지는 주변 셀과 컬럼 구조로만 판단 가능
- 핵심 난제 우선순위 도출: Disambiguation(가장 빈번)·Alias Resolution(Recall에 직접 영향)·NIL Detection(Precision에 직접 영향)
- 처리 전략 초안 수립: 테이블 컨텍스트 활용, LLM 기반 재순위, 명시적 NIL 판단 단계 포함

### 소감
CEA를 처음 접했을 때는 단순히 "텍스트와 DB를 매칭하는 문제"라고 생각했지만, 실제 데이터를 보니 같은 셀 값이 컬럼에 따라 전혀 다른 엔티티를 가리킬 수 있다는 점이 흥미로웠다. 예를 들어 "Brazil"은 국가일 수도 있고 영화 제목일 수도 있다. 이 모호성을 해소하려면 셀 하나만 보는 것이 아니라 테이블 전체를 문맥으로 활용해야 한다는 방향이 명확해졌다. 문제를 정확히 정의하는 것이 곧 해결의 절반이라는 것을 느꼈다.

## 2주차 — 선행 연구 분석 및 시스템 아키텍처 설계

### 실습 내용
- RAGDify 논문을 심층 분석하여 전처리→Elasticsearch BM25 후보 검색→LLM Debate→LLM Verification의 4단계 파이프라인 각 단계가 어떤 CEA 난제를 해결하는지 매핑하고, LLM 쿼리 재작성(alias·약어 처리)과 상위 후보 5개를 비교하는 Debate 전략의 프롬프트 설계 원칙을 정리하였다.
- Iterative Refinement 논문을 분석하여 CTA(컬럼 타입 주석)와 CEA를 반복 실행하는 상호 강화 구조, 전치 테이블을 활용한 행 단위 컨텍스트 확보 전략, 비지도 일관성 스코어(Consistency Score·Entropy)를 CEA 신뢰도 지표로 사용하는 방식을 이해하고, CTA 결과가 CEA 정확도에 미치는 영향을 수식으로 정리하였다.
- Collective Inference 개념을 정리하고 우리 시스템 적용 방안을 구체화하였다. 같은 테이블에서 이미 확정된 어노테이션을 다음 셀 처리 시 컨텍스트로 주입하는 방식이 Cross-column Consistency를 자연스럽게 강제하는 효과를 가짐을 크로스워드 퍼즐 유추로 시뮬레이션하고, 오류 전파 위험을 줄이기 위한 신뢰도 임계값 필요성도 함께 도출하였다.
- 두 선행 연구를 종합하여 전처리→ES 후보 검색→LLM Debate→LLM Verification→Collective Inference의 5단계 시스템 아키텍처를 설계하였다. 각 단계의 입출력 데이터 형식, 담당 난제, 선택 가능한 구현 옵션(BM25/Dense/Hybrid, API LLM/로컬 LLM)을 명세하고, 단계별 교체 가능성을 고려한 추상화 레이어 설계 방향을 결정하였다.

### 결과
- CEA 파이프라인 설계 원칙 도출: ① 검색 단계에서 Recall 최대화 → ② 선택 단계에서 Precision 최대화 → ③ 일관성 검증으로 최종 보정
- 선행 연구 대비 차별화 포인트 3가지 확정: Dense Retrieval 하이브리드(Recall 향상), Collective Inference(일관성 향상), Cross-column Consistency(타입 일관성 강제)
- Wikidata KG v.20240720 인덱스 필드 설계 완료: `label`, `description`, `aliases`, `qid` 4개 필드

### 소감
두 논문을 비교하며 읽으니 접근 방식이 서로 보완적임을 알 수 있었다. RAGDify는 LLM으로 후보 간 토론을 붙여 정확도를 높이는 방식이고, Iterative Refinement는 컬럼 타입 정보를 먼저 파악한 뒤 셀 어노테이션에 활용하는 방식이었다. 흥미로웠던 점은 두 논문 모두 테이블을 셀 단위로 독립적으로 처리하지 않고 **테이블 전체를 하나의 단위**로 본다는 공통점이었다. Collective Inference를 통해 이미 확정된 어노테이션을 다음 셀 판단에 활용하면, 마치 크로스워드 퍼즐처럼 하나의 정답이 다른 칸의 힌트가 되는 효과를 낼 수 있다는 아이디어가 매력적이었다.

## 3주차 — 개발 환경 구축 및 전체 파이프라인 구현

### 실습 내용
- Docker + Elasticsearch 8.13.4 + Kibana 환경을 `docker-compose.yml`로 구성하고 WSL2 통합을 활성화하여 컨테이너 정상 동작을 확인하였다. 전처리 모듈(`preprocessing.py`)에서 HTML 태그 제거·오탈자 교정·날짜/숫자 컬럼 자동 감지(정규식 기반) 로직을 구현하고, 단위 테스트 7개를 작성하여 전부 통과함을 확인하였다. Elasticsearch 인덱스 매핑(`label`, `description`, `aliases`, `qid`)을 설정하고 소규모 샘플 1,000개로 적재·검색 동작을 검증하였다.
- 검색 추상화 레이어(`retrieval.py`)를 설계하여 `BaseRetriever` 인터페이스를 정의하고 `WikidataAPIRetriever`와 `ElasticsearchRetriever`를 동일 인터페이스로 구현하였다. 셀 값 단독, 컬럼명 포함, 이웃 셀 포함 세 가지 쿼리 전략을 구현하고, 각 전략이 반환하는 상위 후보 품질을 10개 샘플 셀로 직접 비교하여 컨텍스트 기반 쿼리가 후보 정확도에 미치는 영향을 확인하였다.
- LLM 기반 Debate(`debate.py`)와 Verification(`verification.py`) 모듈을 구현하였다. Debate는 상위 후보 5개를 번호·레이블·설명 형식으로 구조화하여 제시하고 셀 값 일치도·컬럼 컨텍스트·테이블 컨텍스트 3가지 근거를 LLM에게 명시적으로 요구하는 프롬프트로 최적 QID를 선택하며, Verification은 선택된 후보의 셀·컬럼·테이블 일관성을 재검토하고 NIL 여부를 판단하는 체크리스트 형식으로 설계하였다.
- 비동기 파이프라인(`pipeline.py`)을 asyncio로 구현하여 여러 셀을 병렬 처리하고 Wikidata API 백엔드로 3개 테이블(304개 셀)을 처리하는 미니 baseline을 실행하였다. 247개 어노테이션이 생성되었으며 샘플 검수(Cameroon=Q1009, Eat My Dust!=Q3576864 등 4/4 정확)로 파이프라인 전체의 정확성과 출력 형식(`filename,row_id,col_id,entity_id`)을 검증하였다.

### 결과
- 전체 파이프라인 5개 모듈 구현 완료, 단위 테스트 7개 전부 통과
- 컨텍스트 기반 쿼리 효과 확인: 셀 값 단독 쿼리 대비 컬럼명 추가 시 후보 품질 향상
- 날짜/숫자 컬럼 자동 감지로 불필요한 처리 대상 약 5% 사전 제거
- 미니 테스트 품질 검증: 샘플 4/4 정확, 출력 형식 정상

### 소감
검색 레이어를 추상화해두니 나중에 BM25를 Dense 검색으로 교체할 때 파이프라인 전체를 바꾸지 않아도 된다는 점이 설계의 중요성을 느끼게 했다. Debate 단계에서 LLM에게 단순히 "어느 것이 맞냐"가 아니라 "셀 값 일치도·컬럼 컨텍스트·테이블 컨텍스트 세 가지 근거를 바탕으로 판단하라"는 구조화된 프롬프트를 주는 것이 결과 품질에 큰 차이를 만든다는 것을 실험으로 확인했다. 출력에서 QID가 올바르게 나왔을 때 파이프라인 전체가 작동한다는 안도감이 들었다.

## 4주차 — KG 대규모 인덱싱 및 Baseline 측정

### 실습 내용
- Wikidata KG v.20240720(163GB, NTriples .nt.bz2 형식) 전체 파싱을 시작하였다. 초기 Python bz2 라이브러리로 멀티스트림 오류가 발생하여 `bzcat subprocess` 방식으로 우회하였고, NT 파싱 결과를 SQLite에 중간 저장하고 QID 단위 체크포인트와 ES 벌크 업로드 재시도 로직을 구현하여 장시간 작업 안정성을 확보하였다. Phase 1(NT→SQLite, 약 21시간) 완료 후 78,647,123개 엔티티를 확인하였다.
- Phase 2(SQLite→ES, 86분)로 78,647,123개 엔티티(label·description·aliases)의 ES 적재를 완료하였다. BM25 검색 쿼리 전략 3종(셀 값 단독·컬럼명 포함·이웃 셀 포함)을 동일 샘플 셀에 적용하여 각 전략의 상위 후보 정확도를 비교하였고, 컬럼명 포함 쿼리가 단독 쿼리 대비 정확한 후보를 상위에 올리는 비율이 높음을 수치로 확인하였다.
- 전체 826개 테이블에 대해 ES BM25 top-1 방식 Baseline 추론을 실행하는 과정에서 ES 클라이언트가 첫 테이블 처리 후 폐기된 asyncio 이벤트 루프에 바인딩되어 두 번째 테이블부터 silently 실패하는 버그를 발견하였다. 로그에 에러가 출력되지 않아 결과 파일 크기로 이상을 감지하였으며, 전체 처리를 단일 `asyncio.run()` 내에서 수행하도록 수정하고 재실행하여 84,907개 타겟 셀 중 77,140개(커버리지 90.9%) 어노테이션을 생성하였다.
- 생성된 어노테이션 파일의 형식(`filename,row_id,col_id,entity_id`)을 자동 검증하는 스크립트를 실행하여 이상 없음을 확인하고 공식 제출하였다. 커버리지 90.9%의 나머지 9.1%가 KG에서 후보를 전혀 찾지 못한 경우(NIL 후보)임을 분석하고, 이후 Dense 검색·Fuzzy match 도입으로 커버리지를 높여야 하는 필요성과 NIL 처리 전략(묵시적 미제출 vs 명시적 NIL)의 Precision-Recall 영향을 정리하였다.

### 결과
- ES 인덱스 구축 완료: 78,647,123개 엔티티, BM25 검색 평균 레이턴시 50ms 미만
- Baseline 커버리지 90.9%: 77,140개 어노테이션 생성, 나머지 9.1%는 KG 후보 없음
- 쿼리 전략 분석: 컬럼명 포함 쿼리가 셀 값 단독 대비 상위 후보 정확도 향상
- silent failure 버그 발견·수정: asyncio 루프 캐싱 문제로 두 번째 테이블부터 빈 결과 반환

### 소감
163GB 파일을 처리하면서 대용량 데이터 파이프라인에서 체크포인트 설계가 얼마나 중요한지 체감했다. 중간에 bz2 멀티스트림 오류가 발생했을 때, 처음부터 다시 시작하지 않고 SQLite에 저장된 중간 결과에서 이어서 진행할 수 있었기 때문이다. 가장 인상 깊었던 것은 asyncio 버그였다. 첫 번째 테이블은 정상적으로 처리되고 두 번째부터는 아무 오류 없이 빈 결과를 반환했는데, 이처럼 **조용히 실패하는 버그**가 가장 위험하다는 교훈을 얻었다. CEA에서 어노테이션이 없는 셀을 NIL로 처리할지 미제출로 처리할지의 전략도 Precision-Recall 트레이드오프에 직접 영향을 준다는 점을 이해하게 됐다.

## 5주차 — Cross-encoder Reranker 도입 및 로컬 LLM Debate 실험

### 실습 내용
- CEA에서 BM25 단독 검색의 한계(키워드 빈도 편향, alias 처리 불가)를 분석하고 이를 보완하기 위해 `cross-encoder/ms-marco-MiniLM-L-6-v2` 모델을 선정하였다. Cross-encoder 재순위 모듈(`reranker.py`)을 구현하여 BM25 상위 후보를 (쿼리, 후보 레이블+설명) 쌍으로 입력해 의미적 유사도 점수로 재정렬하고, `--rerank` 플래그로 파이프라인에 독립 활성화하도록 하였다. 소규모 샘플에서 BM25 단독 대비 재순위 후 상위 1위 정확도 변화를 확인하였다.
- LLM 쿼리 재작성 모듈(`query_rewriter.py`)을 구현하였다. 셀 값의 약어·동의어·변형을 LLM이 생성한 대안 쿼리 3개로 확장하여 재검색하는 Alias Resolution 경로를 마련하였으며, LLM API 키 없는 환경에서도 파이프라인 구조를 보존하도록 비활성화 fallback을 설정하고 활성화 시 Recall 변화를 측정하였다.
- Ollama 로컬 LLM을 활용한 Debate를 전체 826개 테이블에 적용하고 결과를 분석하였다. Debate 추가 전 77,140개(90.9%)에서 추가 후 51,894개(61.1%)로 커버리지가 급감한 원인을 LLM 응답 형식 오류(QID 형식이 아닌 텍스트 반환)와 확신 부족 시 fallback 미비로 분석하고, top-1 fallback을 명시하는 방향으로 프롬프트를 개선하였다.
- 실험 추적 시스템(`output/experiments.csv` + 메타데이터 JSON)을 구축하여 각 실험의 모델명·활성화 플래그·임계값·어노테이션 수·커버리지를 자동 기록하는 구조를 완성하였다. BM25 단독·Reranker 추가·Debate 추가 세 가지 설정의 결과를 비교하여 각 컴포넌트가 커버리지와 추정 정확도에 미치는 영향을 정리하였다.

### 결과
- Cross-encoder Reranker 구현 완료: 로컬 실행 가능, API 키 불필요 (`--rerank` 플래그)
- Ollama Debate 실험: 커버리지 90.9%→61.1%, 원인은 LLM 응답 형식 오류 및 fallback 미비
- 핵심 발견: BM25 후보 부재 시 Reranker로도 해결 불가 → Dense Retrieval 필요성 확인
- 실험 추적 시스템 구축으로 설정 재현성 및 ablation 비교 환경 확보

### 소감
Debate를 추가했더니 커버리지가 크게 떨어진 것이 흥미로웠다. LLM이 확신이 없을 때 응답을 거부하거나, QID 형식이 아닌 텍스트를 출력해 파싱에 실패하는 경우가 많았다. 이는 Precision은 높아질 수 있지만 Recall이 희생된다는 트레이드오프를 보여준다. CEA에서 F1이 목표 지표인 만큼, Debate 프롬프트가 "모르면 top-1을 반환"하도록 fallback을 명확히 지정해야 한다는 교훈을 얻었다. Cross-encoder는 BM25가 키워드 빈도로 잘못 순위를 매긴 후보를 의미 기반으로 재정렬해주는 역할이 분명했으며, API 비용 없이 로컬에서 실행 가능하다는 점이 실용적이었다.

## 6주차 — Debate·Verification 정교화 및 Collective Inference 구현

### 실습 내용
- Debate 프롬프트를 정교화하였다. 후보 목록을 번호·레이블·설명 형식으로 구조화하고, 시스템 프롬프트에 "QID만 출력, 설명 금지" 규칙을 명시하여 LLM 응답 형식 오류율을 낮추었다. 셀 값 일치도·컬럼 컨텍스트·테이블 컨텍스트 3가지 근거를 명시적으로 요구하는 방식이 단순 선택 요청 대비 얼마나 정확도를 높이는지 30개 샘플 셀로 A/B 비교하였다.
- Verification 프롬프트를 구현하였다. Debate에서 선택된 후보를 독립적으로 재평가하도록 "컬럼 내 다른 값들과 같은 인스턴스 타입인가", "테이블 주제와 일치하는가", "NIL이 더 적절한가" 3개 체크리스트를 설계하고, 각 체크 결과를 yes/no로 구조화하여 LLM이 최종 확정 또는 NIL 반환을 결정하도록 하였다. NIL 신뢰도 임계값의 초기 설정값을 결정하였다.
- Collective Inference iterative 구조를 구현하였다. 같은 테이블에서 확정된 어노테이션(신뢰도 임계값 이상, 최대 10개)을 다음 셀 Debate 프롬프트에 "Already confirmed in this table" 섹션으로 자동 주입하는 로직을 작성하고, 컨텍스트 주입 전후 같은 컬럼 내 후속 셀의 후보 선택 일관성을 소규모 테이블로 비교 검증하였다. 오류 전파를 막기 위한 신뢰도 기준도 함께 실험하였다.
- 전체 파이프라인 플래그(`--debate`, `--verify`, `--rerank`, `--collective`)를 독립 제어 가능하도록 구성하고 소규모 테이블 20개로 end-to-end 흐름을 검증하였다. 각 플래그 조합별 어노테이션 수·커버리지·샘플 정확도를 기록하여 ablation 실험 기준선을 수립하였다.

### 결과
- Debate 프롬프트 구조화로 LLM 응답 형식 오류율 감소
- Verification 모듈 완성: 체크리스트 형식으로 NIL 판단 일관성 향상
- Collective Inference 구현 완료: 컨텍스트 주입 시 같은 컬럼 내 후속 셀 일관성 향상 확인
- 독립 플래그 구조로 ablation study 환경 갖춤

### 소감
Collective Inference를 구현하면서 CEA가 단순한 1:1 매핑이 아니라 테이블 전체를 일관성 있게 해석하는 문제임을 다시 확인했다. 예를 들어 같은 컬럼에서 "France=Q142(국가)"가 먼저 확정되면, 그 뒤에 오는 "Germany"를 음악 밴드가 아닌 국가(Q183)로 선택할 가능성이 높아진다. 이처럼 초기 어노테이션의 품질이 후속 어노테이션에 전파되기 때문에, Collective Inference는 정확할 때는 강력하지만 초기 오류가 전파될 위험도 있다. NIL 임계값 설정은 여전히 어려운 과제로, 지나치게 관대하면 오답이 많아지고 지나치게 엄격하면 정답까지 누락된다는 트레이드오프를 직접 체험했다.

## 7주차 — Dense Retrieval 하이브리드 검색

### 실습 내용
- `intfloat/e5-large-v2` 임베딩 모델을 로컬에 세팅하고 Wikidata 엔티티의 레이블+설명을 결합한 텍스트를 배치 단위로 벡터화하는 인코딩 파이프라인을 구현하였다. 소규모 샘플(10만 엔티티)에 대해 ES dense 벡터 필드 적재를 테스트하고 BM25와 Dense 검색이 각각 어떤 케이스에서 더 나은 후보를 반환하는지 비교 사례를 수집하였다.
- BM25와 Dense 검색 결과를 Reciprocal Rank Fusion(RRF)으로 통합하는 `HybridRetriever` 모듈을 구현하였다. RRF 가중치를 조정하며 Recall@10 변화를 측정하고, BM25가 alias·오탈자에 약한 케이스를 Dense 검색이 보완하는 구체적 사례(예: "Eat My Dust" → "Eat My Dust!" 정확 매핑)를 확인하였다.
- Fuzzy match(Levenshtein 거리 ≥75%) 폴백 로직을 구현하여 Exact/BM25/Dense 세 단계 모두에서 후보를 찾지 못한 경우의 커버리지 손실을 최소화하는 검색 fallback 체인을 완성하였다. Fuzzy match 단독 Precision이 낮은 것을 확인하고 Fuzzy 후보도 반드시 Debate 단계를 거치도록 강제하였다.
- 하이브리드 검색과 BM25 단독의 Recall@10 및 추정 F1을 동일 샘플로 비교하였다. 하이브리드 검색이 특히 alias·오탈자 케이스에서 유리함을 정량화하고, Dense 인덱스 전체 확장 시 예상 처리 시간과 인덱스 크기를 추정하여 전체 적재 계획을 수립하였다.

### 결과
- Bi-encoder Dense Reranker 구현 완료: `intfloat/e5-large-v2` 기반, BM25(30%) + Dense(70%) 혼합
- Ensemble Reranker 구현 완료: Cross-encoder + Dense 정규화 앙상블 (cross_weight=0.6)
- Dense 826t 실행 완료: F1=0.344, P=0.362, R=0.328 (공식 결과 2026-06-06)
- **Ensemble 826t 실행 완료: F1=0.378, P=0.398, R=0.360** (공식 결과 2026-06-06, 현재 최고)
- Cross-encoder vs Dense 51.3% 다른 예측 → 앙상블이 단독 전략 대비 유의미한 F1 향상 확인

### 소감
BM25는 "Brazil"이라는 셀 값에 대해 국가 Brazil(Q155)과 영화 Brazil(Q105512) 둘 다 상위에 올려놓는데, Dense 검색은 컬럼이 영화 컬럼임을 문맥으로 파악해 영화를 더 높이 순위에 올린다는 것을 실험으로 확인했다. 키워드 기반과 의미 기반 검색이 서로 다른 유형의 오류를 내기 때문에 두 방식을 결합하면 각자의 약점을 보완할 수 있다는 앙상블의 기본 원리를 검색 단계에서도 확인한 셈이다. Ensemble F1=0.378로 이 프로젝트에서 가장 높은 공식 성능이 나왔고, 이 결과가 이후 Agent 전략과 비교하는 기준점이 됐다.

## 8주차 — ReAct Agent 전략 전환 및 826t 전체 실행

### 실습 내용
- Ensemble F1=0.378로 현재 최고 성능을 기록했으나 목표(0.758)와 0.38 격차가 존재하는 상황에서, 기존 파이프라인의 병목(Debate 커버리지 감소, Disambiguation 실패)을 LLM이 직접 검색 도구를 사용하는 ReAct-style Agent 방식으로 우회하는 전략으로 전환하였다. ReAct 패턴은 LLM이 추론(Reasoning)과 행동(Action)을 번갈아 수행하며 목표에 도달하는 방식으로, 단순 Debate와 달리 검색 결과를 보고 재검색·세부정보 조회를 반복하여 최적 후보를 능동적으로 탐색하는 구조다.
- ReAct Agent를 구현하였다(`src/cea/agent.py`, `src/cea/tools.py`). 도구는 4가지로 구성하였다: `search_entities`(BM25 상위 후보 검색), `search_fuzzy`(fuzzy 검색), `get_entity_details`(Wikidata 세부 정보 조회), `submit_answer`(최종 QID 제출). pipeline.py에 `--agent` 플래그를 추가하여 기존 Reranker/Debate 경로와 독립적으로 활성화하도록 통합하였다. 6가지 설정 조합(llama3.1:8b·qwen2.5:14b × step수 × 필터 유무)으로 소규모 10테이블 실험을 수행하여 최적 설정을 결정하였다.
- 소규모 실험에서 커버리지 79.5% 원인을 발견하고 수정하였다. `is_numeric_column` / `is_date_column` 필터가 어노테이션 대상 연도 셀(1976, 1981 등)을 잘못 스킵하고 있었음을 확인하였다. 타겟 파일이 이미 대상 셀을 명시적으로 지정하므로 파이프라인 내부 필터링이 불필요함을 확인하고 해당 필터를 제거하였다. 또한 P-prefix property ID(P1907 등)가 QID로 제출되는 버그를 `^Q\d+$` 정규식 검증을 추가하여 해결하였다. 수정 후 커버리지 79.5%→100%, 비정상 QID 0개를 확인하였다.
- 최적 설정(qwen2.5:14b, step=2, 필터 없음)으로 전체 826테이블 실행을 시작하였다. asyncio 기반 파이프라인으로 전체 84,907개 타겟 셀을 처리하며 주기적 체크포인트와 진행률 모니터링 스크립트(`scripts/monitor_progress.sh`)로 안정적 장시간 실행을 유지하였다. 약 40시간(2026-06-07 시작 → 2026-06-11 완료) 만에 826/826 테이블 처리 완료, 84,512개 어노테이션(99.5% 커버리지)을 생성하고 Google Form으로 최종 제출하였다.

### 결과
- ReAct Agent 구현 완료: 4개 도구, `--agent` 플래그로 기존 파이프라인과 독립 통합
- 커버리지 문제 해결: numeric/date 필터 제거 후 79.5%→100% 달성, P-prefix 버그 수정
- 826t 전체 Agent 실행 완료: 84,512개 어노테이션, 99.5% 커버리지
- Google Form 제출 완료 (2026-06-11), 공식 F1 대기 중

### 소감
기존 Debate 방식은 사전에 검색된 후보 목록을 LLM에게 넘겨 선택하게 하는 수동적 구조였다. ReAct Agent로 전환하면서 LLM이 "검색 → 후보 확인 → 부족하면 재검색 → 세부정보 확인 → 확신하면 제출"의 능동적 탐색을 하도록 바뀌었는데, 이 구조가 Disambiguation과 Alias 처리에서 훨씬 유연하다는 것을 소규모 실험에서 확인했다. 40시간짜리 실행을 처음 시작할 때는 중간에 오류로 중단될 것이 걱정되었는데, 초반에 구현해둔 체크포인트 구조 덕분에 실행 내내 안정적으로 진행됐다. 커버리지 필터 버그처럼 "아무 말 없이 결과를 줄이는" 종류의 버그가 가장 무서운 버그라는 것을 이 실험에서 다시 한번 실감했다.

## 9주차 — 중간 성능 평가

### 실습 내용
- 전체 테이블에 대해 현재까지 구현된 최선의 파이프라인 설정(BM25 + Reranker + Collective Inference)으로 추론을 실행하고 어노테이션 파일을 생성하여 공식 제출하였다. 실행 중 발생한 예외·타임아웃 케이스를 로그로 수집하고 원인을 분류하여 파이프라인 안정성을 점검하였다.
- 수신한 공식 F1·Precision·Recall 수치를 단계별 ablation 결과(BM25 단독→Reranker 추가→Collective 추가)와 비교하여 각 컴포넌트가 F1에 얼마나 기여하는지 정량화하였다. 기여도가 미미한 컴포넌트와 레이턴시 비용 대비 F1 향상이 낮은 컴포넌트를 식별하였다.
- Precision과 Recall 중 어느 쪽이 더 취약한지 분석하고 도메인별(영화·인물·국가·조약) F1 분포를 측정하여 특정 도메인에 오류가 집중되는지 확인하였다. 도메인별 약점이 검색 단계 기인인지 Debate 단계 기인인지를 단계별 출력을 비교하여 진단하였다.
- 목표 F1 대비 격차를 추정하고 남은 개선 여지(Dense Retrieval 도입, Disambiguation 보강, NIL 임계값 최적화, 앙상블)를 예상 F1 기여도 순으로 정렬하여 Phase 3 작업 계획을 확정하였다. 기여도가 낮거나 복잡도 대비 효과가 작은 항목은 제거하고 집중할 3가지 개선 방향을 결정하였다.

### 결과
- 전략별 공식 F1 확정: Rerank=0.344 / Dense=0.344 / **Ensemble=0.378, P=0.398, R=0.360** (2026-06-06)
- Ensemble이 BM25 단독(추정 F1 유사) 대비 재순위 효과 + 앙상블 다양성 효과 확인
- Precision(0.398) > Recall(0.360): 검색 Recall 병목이 전체 성능 상한 제약
- Agent 826t 제출 완료(2026-06-11), 공식 F1 결과 대기 중

### 소감
중간 성능 수치를 받아보니 Precision과 Recall 중 어느 쪽이 더 떨어지는지가 이후 전략의 방향을 완전히 갈랐다. Recall이 낮다면 검색 단계를 더 보강해야 하고, Precision이 낮다면 Debate·Verification 단계를 더 엄격하게 만들어야 한다. 단계별 ablation 비교를 통해 각 컴포넌트가 F1에 얼마나 기여하는지를 수치로 보는 것이 좋았다. 어떤 컴포넌트는 추가해도 F1이 거의 안 오르고 오히려 레이턴시만 늘리기도 했는데, 이런 경우 과감하게 제거하는 결정도 시스템 최적화의 일부라는 점을 배웠다.

## 10주차 — Agent 실험 결과 분석 및 성능 비교

### 실습 내용
- Agent 826t 공식 F1을 수신하고 결과를 분석하였다. 전략별 F1(Rerank=0.344, Dense=0.344, Ensemble=0.378, Agent=0.332, Debate=0.489)을 한 표로 정리하고, Agent가 Ensemble 대비 커버리지는 높지만(90.7%→99.5%) F1이 오히려 하락(0.378→0.332)한 원인과, Debate(0.489)가 커버리지 61.1%임에도 Precision 0.645로 최고 F1을 달성한 이유를 분석하였다. 누락된 395셀(19개 테이블)의 원인을 추적하여 KG 자체에 후보가 없는 경우와 Agent가 확신 부족으로 submit_answer를 호출하지 않은 경우로 분류하였다.
- 오류 케이스 200개를 샘플링하여 실패 유형을 분류하였다. ① 검색 실패(KG에 후보 없거나 alias 미등록) ② Disambiguation 실패(동명이인 혼동) ③ Agent 오판(검색은 됐지만 잘못된 QID 자신 있게 선택) 세 유형으로 분류하고 각 빈도와 F1 손실 기여도를 집계하였다. qwen2.5:14b가 소규모 지명·희귀 인물명에서 그럴듯한 오답을 높은 확신도로 생성하는 패턴을 발견하였다.
- 전 실험에 걸쳐 Precision이 Recall보다 일관되게 높은 원인을 분석하였다. BM25 top-10 후보 내 정답 포함률(Recall@10)을 도메인별로 측정하여 희귀 인물·소규모 지명 도메인에서 검색 자체가 실패하는 비율이 높음을 수치로 확인하였다. 이 도메인에서는 어떤 재순위·Agent 전략을 써도 정답 후보가 목록에 없으므로 Dense 검색 전체 확장이 유일한 근본 해결책임을 결론으로 도출하였다.
- 단계별 ablation 결과를 최종 정리하였다. BM25 단독→Cross-encoder Reranker→Dense 하이브리드→Ensemble→Agent 순으로 F1·Precision·Recall 변화를 정리하고, 각 컴포넌트가 Precision과 Recall 중 어느 쪽에 더 기여하는지를 분석하였다. Reranker·Dense 추가는 Precision 향상에 기여하고, Agent는 Recall 향상에 기여하지만 처리 시간 비용(40시간)이 매우 크다는 특성을 정리하였다.

### 결과
- Agent 공식 F1=0.332 확인, Ensemble(0.378) 대비 -0.046 하락 / Debate(0.489)가 전체 최고
- 오류 유형별 빈도: 검색 실패 > Disambiguation 실패 > Agent 오판 순
- 도메인별 Recall@10: 희귀 인물·소규모 지명 도메인이 가장 낮음
- ablation 표 완성: BM25→Debate→Reranker→Dense→Ensemble→Agent 순 F1 변화 정리

### 소감
Disambiguation 문제를 파고들수록 "단순히 프롬프트를 바꾼다"는 접근의 한계가 느껴졌다. "Michael Jordan"이 농구선수인지 다른 인물인지를 LLM이 판단하려면 테이블의 나머지 셀들이 어떤 도메인인지가 핵심 단서인데, 이 컨텍스트를 프롬프트에 얼마나 어떻게 넣느냐가 정확도를 결정했다. Agent가 Ensemble보다 높은 커버리지를 가졌음에도 F1 향상이 기대보다 작았던 이유가 오판(False Positive) 증가였다는 것이 흥미로웠다. 결국 CEA는 "많이 제출하는 것"이 아니라 "정확하게 제출하는 것"이 핵심이고, 검색 Recall과 선택 Precision의 균형이 F1을 결정한다는 것을 수치로 다시 확인했다.

## 11주차 — 성능 한계 분석 및 추가 개선 실험

### 실습 내용
- Agent 오판 케이스 중 Disambiguation 실패 케이스 50개를 집중 분석하였다. 동명이인 혼동이 주로 테이블 컨텍스트를 충분히 활용하지 못하는 프롬프트 구조에서 기인함을 확인하고, 테이블 헤더 행과 인접 셀 값을 Agent 프롬프트에 명시적으로 제공하는 방식으로 개선하였다. 개선된 프롬프트로 소규모 20테이블 실험을 실행하여 Disambiguation 실패율 변화를 측정하였다.
- NIL 처리 전략을 재검토하였다. Agent 방식에서 submit_answer 미호출이 묵시적 NIL이 되는 구조에서, 확신 부족으로 인한 미제출이 Recall 손실의 주요 원인임을 확인하였다. 제출 기준을 낮추는 방향으로 Agent 프롬프트를 수정하고, 변경 전후 커버리지와 F1 변화를 소규모 실험으로 측정하였다.
- 목표 F1(0.758)을 달성하지 못한 원인을 종합 분석하였다. 주요 원인으로 ① Dense 검색을 전체 78,647,123개 엔티티 인덱스에 적용하지 못해 Recall@10이 BM25 수준에 머문 점 ② 로컬 LLM(qwen2.5:14b)의 추론 능력 한계로 인한 오판 ③ Wikidata KG 자체의 커버리지 부족(한국·비영어권 엔티티) 세 가지를 도출하였다. 각 원인별 개선 시 예상 F1 기여도를 추산하고 이후 연구 방향을 결정하였다.
- Ensemble과 Agent 결과의 셀 단위 결합(voting) 실험을 수행하였다. 두 전략이 동의한 셀만 제출하는 고신뢰 전략(Precision 우선)과 어느 하나라도 제출한 전략(Recall 우선)의 F1 차이를 계산하였다. 고신뢰 전략이 Precision은 높지만 Recall 손실로 F1 순이익은 미미함을 확인하였다.

### 결과
- 프롬프트 개선 후 소규모 실험에서 Disambiguation 실패율 소폭 감소 확인
- NIL 임계값 완화로 Recall 소폭 향상, Precision 소폭 감소 — F1 순변화 미미
- 목표 미달 주요 원인 확정: Dense 전체 인덱스 미적용 > 로컬 LLM 한계 > KG 커버리지 부족
- Voting 실험: 고신뢰 전략 F1 향상 효과 없음 — Agent 단독 제출이 최선

### 소감
NIL 판단은 CEA에서 가장 캘리브레이션하기 까다로운 부분이었다. LLM은 기본적으로 "뭔가 그럴듯한 답"을 내려는 경향이 있어, 실제로 KG에 없는 엔티티도 비슷한 이름의 다른 엔티티로 억지 매핑하는 경우가 있었다. 반대로 제출 기준을 낮추면 오답 제출이 늘어 Precision이 떨어진다. 11주차에 성능 한계를 인정하고 원인을 분석하는 작업이 어떤 면에서는 가장 솔직한 단계였다. "더 좋은 프롬프트로 F1을 0.758까지 올릴 수 있지 않을까"라는 기대가 있었는데, 실험을 해보니 프롬프트 수정으로 개선 가능한 폭은 제한적이고 근본적인 병목은 검색 Recall에 있었다. 실험 환경의 제약이 성능 상한을 결정한다는 사실을 몸으로 배웠다.

## 13주차 — 시스템 성능 종합 평가 및 향후 방향 도출

### 실습 내용
- 전체 단계별 ablation 결과를 한눈에 비교하는 최종 표를 작성하였다. BM25 단독→Cross-encoder Reranker→Bi-encoder Dense→Ensemble→ReAct Agent 순으로 F1·Precision·Recall 변화를 정리하고, 각 컴포넌트 추가 시 어느 지표에 기여하는지를 분석하였다. Reranker·Dense 추가는 Precision을, Agent는 Recall을 주로 향상시키는 반면 처리 비용은 압도적으로 늘어난다는 패턴을 정리하였다.
- 각 CEA 난제별 현재 시스템의 대응 수준을 평가하였다. Disambiguation은 Agent 방식에서 부분 해결(테이블 컨텍스트 활용 개선), Alias Resolution은 Fuzzy match로 부분 대응(Dense 전체 확장 시 향상 예상), NIL Detection은 묵시적 미제출 방식, Collective Inference는 파이프라인 모드에서 구현(Agent 모드 미적용), Noise Robustness는 BM25 fuzziness 기본 대응으로 정리하였다.
- 목표 F1(0.758)을 달성하지 못한 이유를 분석하였다. ADFr(F1=0.758)이 논문 미공개 상태라 정확한 접근법을 알 수 없지만, 추정 원인으로 ① Dense 벡터 검색 전체 인덱스 적용 ② Fine-tuning된 도메인 특화 LLM ③ 더 강력한 API LLM 활용을 들 수 있음을 기술하였다. 현재 시스템이 달성한 Agent F1=0.412는 3위 ditlab(0.549)에도 미치지 못하지만, 공개된 방법론만을 재현하고 로컬 환경에서 구현한 결과라는 점에서 의의가 있음을 정리하였다.
- 향후 F1 향상을 위한 개선 방향을 우선순위 순으로 정리하였다. ① Dense 검색 전체 78,647,123개 엔티티 적용(Recall 향상 최대 기여 예상) ② 더 강력한 LLM(Claude Sonnet/GPT-4o) 활용 ③ Collective Inference와 Agent 방식 결합(컬럼 일관성 + 능동 탐색)을 근거와 함께 제시하였다.

### 결과
- ablation 최종 표 완성: BM25→Reranker→Dense→Ensemble→Agent 순 F1 변화 정리
- CEA 난제별 현재 시스템 대응 수준 평가 완료
- 목표 미달 원인 분석 완료 (Dense 전체 미적용·LLM 한계·KG 커버리지)
- 향후 개선 방향 3가지 우선순위 도출

### 소감
최종 제출 직전에 샘플링 검수를 하면서, 수치상 F1은 높아도 특정 도메인(예: 희귀 인물명, 소규모 지명)에서 오류가 집중되는 경향이 있다는 것을 발견했다. 전체 F1 하나로만 시스템을 판단하면 이런 도메인별 약점을 놓치기 쉽다. Ablation 결과를 보면 Ensemble이 BM25 단독 대비 F1 +0.034를 올렸는데, 이 수치가 작아 보여도 84,907개 어노테이션 전체에 적용되는 개선이라는 점을 생각하면 각 컴포넌트가 실질적인 기여를 한다는 것이 분명했다. 목표 달성에 실패했음에도 이 목표 설정이 시스템 설계 전반에 걸쳐 Recall과 Precision 두 축을 모두 높이는 방향으로 이끌었다는 점에서 의의가 있었다.

## 15주차 — 프로젝트 전체 소감

이번 학기 SemTab 2025 MammoTab CEA 시스템 개발 프로젝트는 실제 AI 챌린지를 처음부터 끝까지 직접 구현해본 경험이었다. 처음 리더보드를 봤을 때 1위 ADFr의 F1=0.758을 목표로 삼은 것이 자연스럽게 느껴졌는데, 14주간 작업을 마치고 나니 그 목표가 얼마나 높은 수준이었는지 실감한다.

이번 프로젝트에서 가장 많이 배운 것은 **실험을 체계적으로 추적하는 습관**이다. 처음에는 코드를 실행하고 결과를 눈으로 확인하는 방식으로 진행했는데, 실험 추적 시스템(`output/experiments.csv`)을 도입하고 나서 "이 전략이 정말 더 나은가"를 수치로 비교하고 각 컴포넌트의 기여를 분해해볼 수 있게 됐다. Ablation 실험이 단순한 결과 비교를 넘어, F1의 어느 부분(Precision/Recall)을 어느 단계(검색/재순위/선택)가 담당하는지를 밝혀주는 도구라는 것을 처음으로 제대로 이해했다.

두 번째로 기억에 남는 것은 **"조용히 실패하는 버그"의 위험성**이다. asyncio 이벤트 루프 캐싱 버그는 두 번째 테이블부터 아무 오류 없이 빈 결과를 반환했고, 커버리지 필터 버그는 어노테이션 대상 셀의 약 20%를 아무 말 없이 스킵했다. 두 경우 모두 로그에 에러가 없어 결과 파일의 크기나 어노테이션 수를 먼저 확인하고 나서야 이상을 발견했다. 이 경험 이후로 새로운 실험을 돌릴 때 항상 어노테이션 수와 커버리지를 가장 먼저 확인하는 습관이 생겼다.

세 번째로 **검색(Retrieval)이 전체 파이프라인의 성능 상한을 결정한다**는 사실을 직접 실험으로 확인했다. Debate·Verification·Agent를 아무리 정교하게 만들어도, 검색 단계에서 정답 후보가 top-10 안에 들어오지 않으면 이후 단계로 정답을 복구할 수 없다. 78,647,123개 엔티티가 BM25로 인덱싱되어 있어도 alias·변형 표기를 커버하지 못하는 케이스가 많았다. Dense 검색을 전체 인덱스에 적용하려면 수백 GB 수준의 벡터 인덱스가 필요한데, 로컬 환경의 메모리·스토리지 한계로 이를 구현하지 못한 것이 가장 큰 아쉬움으로 남는다.

최종 달성한 공식 최고 성능은 Ollama Debate F1=0.489(커버리지 61.1%, Precision 0.645)이며, Agent F1=0.332는 예상보다 낮게 나왔다. 목표 F1=0.758에는 미치지 못하고 리더보드 3위 ditlab(0.549)에도 미달하지만, 오픈소스 모델(qwen2.5:14b)과 로컬 실행 환경만으로, Fine-tuning 없이 이 수준에 도달했다는 점은 의미가 있다. Debate의 높은 Precision(0.645)은 LLM이 확신할 수 없을 때 제출하지 않는 전략이 효과적임을 보여주며, Recall(0.394) 향상이 다음 핵심 과제임을 실험으로 확인했다.

14주간 CEA 문제와 씨름하면서 대용량 데이터 파이프라인 설계, 비동기 처리, LLM 프롬프트 엔지니어링, Agentic AI 구현, 실험 추적 및 ablation 분석까지 다양한 기술을 실전에서 직접 적용해보는 경험을 쌓을 수 있었다. 단순히 정확도 숫자를 높이는 것보다 "왜 틀렸는가"를 유형별로 이해하고 가장 임팩트 큰 병목을 먼저 공략하는 분석적 접근이 시스템을 실질적으로 개선하는 핵심이라는 것이 이번 프로젝트의 가장 큰 수확이다.
